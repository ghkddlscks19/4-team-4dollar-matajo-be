### 1.2 성능 목표 설정 근거 (비즈니스 맥락)

#### 당근마켓 벤치마킹

마타조는 당근마켓을 모티브로 한 로컬 마켓플레이스 서비스입니다. 당근마켓의 전략 변화를 참고하여 성능 목표를 설정했습니다.

**당근마켓 현황 (2024-2025)**:
| 지표 | 수치 | 비고 |
|------|------|------|
| 누적 가입자 | 4,300만 명 | |
| MAU | 2,000만 명 | 성장 정체 (의도적) |
| WAU | 1,300~1,400만 명 | |
| DAU | 600~900만 명 (추정) | **핵심 집중 지표** |
| 사용 시간 증가율 | +12~13%/년 | 두 자릿수 성장 |

**당근마켓의 전략적 시사점**:
- MAU 확대보다 **DAU/체류시간** 증가에 집중
- 광고 수익 = DAU × 체류시간 × 광고단가
- 부가 서비스(알바, 부동산, 동네생활)로 "매일 들어올 이유" 창출

#### 마타조 서비스 목표 규모

당근마켓의 약 **1/100~1/150 규모**를 초기 목표로 설정:

| 지표 | 당근마켓 | 마타조 (목표) | 비율 |
|------|----------|--------------|------|
| DAU | 600~900만 | **5만** | 1/150 |
| 피크 동시접속 | 30~50만+ | **5,000** | 1/100 |
| 목표 TPS | - | **2,000** | - |

이는 특정 지역(구/동 단위) 또는 초기 스타트업 서비스로서 현실적인 목표입니다.

#### 동시 접속자 5,000명 산출 근거

```
DAU: 50,000명
평균 체류시간: 10분
서비스 운영시간: 18시간 (06:00~24:00)
피크 시간대 집중률: 3배

평균 동시접속 = 50,000 × (10분/60분) / 18시간 × 60분
             = 50,000 × 0.167 / 1,080분 × 60분
             ≈ 463명

피크 동시접속 = 463 × 3 ≈ 1,400명 (이론값)
안전 마진 적용 (3.5배) = 1,400 × 3.5 ≈ 5,000명
```

## 2. 성능 기준 (Performance Criteria)

### 2.1 실시간 채팅 시스템의 성능 기준 정의

실시간 채팅 서비스는 **사용자 체감 품질(QoE)**이 핵심입니다. 아래는 업계 표준과 사용자 경험 연구를 기반으로 설정한 성능 기준입니다.

#### P95 (95th Percentile) 레이턴시란?

```
P95 = 전체 요청 중 95%가 이 시간 이내에 완료됨
```

**왜 P95를 사용하는가?**
- **평균(Avg)**: 극단값에 민감하여 실제 사용자 경험을 반영하지 못함
- **P50 (중간값)**: 절반의 사용자 경험만 반영
- **P95**: 대부분의 사용자(95%) 경험을 보장하면서 합리적인 목표 설정
- **P99**: 더 엄격하지만, 달성하기 어려워 초기 목표로는 부적합

### 2.2 WebSocket 실시간 채팅 성능 기준

| 메트릭 | 목표 값 | 근거 |
|--------|---------|------|
| **연결 성공률** | ≥ 99.9% | 0.1% 미만 실패율 유지 |
| **연결 시간 P95** | ≤ 3초 | 3초 이상 대기 시 33% 사용자 이탈 (Google 연구) |
| **메시지 레이턴시 P50** | ≤ 100ms | 실시간 대화 느낌 유지 |
| **메시지 레이턴시 P95** | ≤ 300ms | 인간이 지연을 인지하는 임계점 |
| **메시지 레이턴시 P99** | ≤ 500ms | 극단적 상황에서도 0.5초 이내 |
| **메시지 전달률** | ≥ 99% | 메시지 유실 최소화 |

#### 레이턴시 기준의 과학적 근거

```
0~100ms   : 즉각적 반응 (사용자가 지연 인지 못함)
100~300ms : 약간의 지연 인지 가능하나 수용 가능
300~500ms : 명확한 지연 인지, 불편함 시작
500ms+    : "느리다"는 인식, 사용자 경험 저하
1000ms+   : 시스템 장애로 인식 가능
```

### 2.3 REST API 성능 기준

| 메트릭 | 목표 값 | 용도 |
|--------|---------|------|
| **TPS (Transactions Per Second)** | ≥ 2,000 | 초당 처리 요청 수 (기존 500에서 상향) |
| **API 성공률** | ≥ 99% | 오류율 1% 미만 |
| **응답 시간 P95** | ≤ 500ms | 95% 요청이 0.5초 이내 완료 |
| **응답 시간 P99** | ≤ 1000ms | 99% 요청이 1초 이내 완료 |

#### TPS (Throughput) 계산 방법

```
TPS = 총 요청 수 / 테스트 시간(초)

예시: 10분간 1,200,000건 처리 → 1,200,000 / 600 = 2,000 TPS
```

**TPS 목표 설정 근거**:
- 동시 사용자 5,000명
- 평균 요청 간격: 2.5초 (채팅방 목록 조회, 메시지 로드 등)
- 예상 TPS: 5,000 / 2.5 = 2,000 TPS
- 버퍼 고려 시 목표: 2,000~3,000 TPS

## 3. 테스트 시나리오

### 3.1 WebSocket 부하 테스트

```javascript
// 점진적 부하 증가 패턴 (Ramp-up)
stages: [
  { duration: '1m', target: 500 },    // 워밍업
  { duration: '2m', target: 1500 },   // 증가
  { duration: '2m', target: 2500 },   // 증가
  { duration: '3m', target: 5000 },   // 목표 도달 (5,000명)
  { duration: '5m', target: 5000 },   // 안정성 검증 (중요!)
  { duration: '2m', target: 2500 },   // 감소
  { duration: '1m', target: 0 },      // 정리
]
// 총 테스트 시간: 약 16분
```

**시나리오 설명**:
1. **워밍업 (1분)**: JIT 컴파일, 커넥션 풀 준비
2. **부하 증가 (7분)**: 단계적으로 5,000 VUs까지 증가
3. **안정성 유지 (5분)**: 피크 부하에서 시스템 안정성 검증
4. **정리 (3분)**: Graceful shutdown 검증

**각 VU(가상 사용자)의 행동**:
- WebSocket 연결 → STOMP CONNECT
- 채팅방 구독 (5,000개 방에 분산)
- 메시지 10개 전송 (2~5초 간격)
- 읽음 상태 업데이트 3회
- 30초 연결 유지 후 종료

### 3.2 REST API 부하 테스트

**테스트 엔드포인트**:
| API | 설명 | 예상 부하 |
|-----|------|----------|
| `GET /api/chats` | 채팅방 목록 조회 | 높음 (메인 화면) |
| `GET /api/chats/{roomId}/message` | 메시지 조회 | 중간 (채팅방 입장 시) |

## 7. 트러블슈팅

### 7.1 일반적인 문제

| 증상 | 원인 | 해결 방법 |
|------|------|----------|
| 연결 성공률 < 90% | 서버 리소스 부족 | Heap 메모리 증가, 커넥션 풀 조정 |
| P95 레이턴시 > 1초 | DB 쿼리 병목 | N+1 문제 해결, 인덱스 추가 |
| 메시지 유실 | WebSocket 버퍼 오버플로우 | 메시지 큐 도입, 배치 처리 |
| OOM 에러 | 메모리 누수 | 세션 정리 로직 확인, GC 튜닝 |

### 7.2 k6 테스트 관련

```bash
# 로그 레벨 조정
k6 run --log-output=stdout --log-format=json script.js

# VUs 수 오버라이드
k6 run --vus=100 --duration=5m script.js

# 특정 시나리오만 실행
k6 run --scenario=gradual_load script.js
```
